TokenSpeed Runtime Design

LoRA Serving Implementation

This document describes the current LoRA implementation in the working branch: how adapter names and ids map to GPU slots, how CPU and GPU eviction work, how dense and MoE LoRA weights are packed, and why the CUDA graph path remains stable across dynamic adapters.

Overview

TokenSpeed treats LoRA as a runtime-owned side path. Base model layers keep their normal linear and MoE kernels. When a request uses an adapter, the runtime resolves that request's lora_id to a GPU slot, writes per-step metadata into persistent tensors, and the model layers add LoRA deltas in place.

🐍 Python Runtime

LoraManager owns adapter identity, GPU/CPU/disk pool eviction, weight sharding, and CUDA-graph-stable LoraBatchInfo tensors. Model layers (qwen3.py, qwen3_5.py) call apply_qkv_lora, apply_o_lora, apply_gate_up_lora, and apply_down_lora after each base projection.

⚡ Kernels (tokenspeed-kernel)

Triton kernels in ops/lora/triton/ handle dense attention and MLP LoRA (shrink + expand, prefill SGMV, decode grouped). MoE LoRA lives in ops/moe_lora/: flat per-expert kernels avoid gather copies; a secondary CUDA stream overlaps A-shrink with the base MoE GEMM.

🗂 Scheduler (C++)

Each request carries a lora_id. The KV prefix cache gives every adapter its own virtual radix-tree root so prefix reuse is isolated per adapter. EvictLoraNamespace(lora_id) prunes the tree on adapter unload.

Loadadapter path → CPU cache
LoraManager.load_adapter()Registers name/id, stores durable disk path, warms CPU cache. (Python)
Schedulerequest ids → adapter ids
prepare_loras()Promotes missing adapters to GPU slots, writes LoraBatchInfo. (Python + Scheduler)

Example User Flow

The following shows a typical lifecycle. The adapter starts on disk; the user registers it once, then references it by name on every request. The engine handles all GPU/CPU eviction automatically.

Python API

from tokenspeed.runtime.entrypoints.engine import Engine

engine = Engine(
    model="Qwen/Qwen3-30B-A3B",
    enable_lora=True,
    max_loras=4,          # max adapters resident on GPU simultaneously
    max_loras_cpu=16,     # max adapters held in CPU memory
    max_lora_rank=64,
)

# Step 1 — Register the adapter (weights stay on disk until needed).
engine.load_lora_adapter("my_adapter", adapter_path="/models/my_lora")

# Step 2 — Use the adapter by name in any generate call.
response = engine.generate(
    prompt="What is the capital of France?",
    sampling_params={"max_new_tokens": 64},
    lora_name="my_adapter",   # reference the registered name
)

# Step 3 — Base-model requests need no lora_name.
response = engine.generate(
    prompt="Summarise this document.",
    sampling_params={"max_new_tokens": 128},
    # lora_name omitted → base model, zero LoRA overhead
)

# Step 4 (optional) — Unload frees the GPU slot and evicts the adapter's
# cached KV pages from the prefix cache. Weights remain on disk.
engine.unload_lora_adapter("my_adapter")

HTTP — OpenAI-compatible endpoints

The same lifecycle applies over the HTTP API. Register adapters once at server startup (or via a management call), then pass "model" as the adapter name on every request.

curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "my_adapter",
    "messages": [{"role": "user", "content": "What is the capital of France?"}],

# /v1/completions  — same "model" field selects the adapter
curl http://localhost:8000/v1/completions \
  -H "Content-Type: application/json" \
  -d '{

# Base model — use the original model name (no LoRA overhead)
curl http://localhost:8000/v1/chat/completions \

    "max_tokens": 128
  }'
Assumption: the adapter directory must exist on disk before the server starts or before calling load_lora_adapter(). The engine records the path at registration time so it can reload weights from disk if CPU pressure evicts them from memory. A name that was never registered will raise an error — it does not fall back to the base model silently.

Naming

This section defines the vocabulary used throughout the LoRA implementation. Each identifier — from the user-facing adapter name to the ephemeral GPU slot — has a distinct role and lifetime, and keeping them separate prevents subtle bugs when adapters are evicted, reloaded, or shared across requests.

Key files — python/tokenspeed/runtime/lora/lora_manager.py (added) · python/tokenspeed/runtime/engine/io_struct.py (added) · python/tokenspeed/runtime/engine/input_processor.py (added)
NameMeaningWhere it lives
name Stable user-facing adapter name or alias, such as "password_adapter". This is the value requests should select after registration. LoraManager._name_to_id, _adapter_paths, CPU/GPU LRU maps.
lora_name Canonical request/API selector. It must be the name of an adapter that was already loaded via load_lora_adapter(). Request schema and input processing before lookup in LoraManager.
adapter_path / load-time path Durable filesystem path to the adapter directory or safetensors file. Every registered adapter needs one so CPU eviction can reload weights from disk. LoraManager._adapter_paths, LoraCpuCache.adapter_paths.
lora_id Runtime integer id assigned at registration time. Request scheduling carries this id. _name_to_id, _id_to_name, request metadata.
slot GPU-resident real adapter slot. Valid slots are 0..max_loras-1; base/no-LoRA is NO_LORA_SLOT = -1 in batch metadata. _slot_to_name, _name_to_slot, LoraBatchInfo.weight_indices.
rank LoRA rank used by the adapter. For 3D MoE tensors, rank is dimension 1 of lora_A. _lora_ranks, _slot_ranks, per-slot buffer slices.
scaling lora_alpha / r from adapter_config.json, or 1.0 fallback. _scalings, _slot_scalings, kernel multiply.
segment One contiguous run of tokens using one adapter slot. Current path uses one segment per request. seg_lens, seg_indptr, weight_indices.
Important distinction: adapter_path is the disk source of truth used when the adapter is loaded or reloaded. Request-time lora_name selects an already loaded adapter. lora_id is stable while the adapter remains registered. slot is temporary and may change after GPU eviction and reload.

Data Model

This section describes the in-memory structures that track adapter weights and residency state. AdapterWeights holds parsed tensors keyed by layer and module name; the registration and residency dictionaries give the manager fast O(1) lookup for every scheduling decision.

Key files — python/tokenspeed/runtime/lora/lora_batch.py (added) · python/tokenspeed/runtime/lora/lora_manager.py (modified) · python/tokenspeed/runtime/execution/context.py (modified)

AdapterWeights

Parsed adapter weights use this logical shape: 

AdapterWeights = {
  layer_id: {
    module_name: (lora_A, lora_B),
  }
}

Dense modules use names like q_proj, o_proj, gate_proj, up_proj, and down_proj.

2D MoE per-expert modules use names like experts.7.gate_proj. 3D MoE modules use experts.w1, experts.w2, and experts.w3.

Registration State

_name_to_id:    dict[str, int]        # user name -> stable lora_id
_id_to_name:    dict[int, str]        # stable lora_id -> user name
_adapter_paths: dict[str, str]        # user name -> durable adapter directory

Residency State

_cpu_cache:          dict[str, AdapterWeights]    # parsed host weights
_cpu_lru:            OrderedDict[str, None]        # CPU eviction order
_name_to_slot:       dict[str, int]                # GPU-resident name -> slot
_slot_to_name:       list[str | None]              # slot -> GPU-resident name
_gpu_lru:            OrderedDict[str, None]        # GPU LRU eviction order
_active_names:       set[str]                      # adapters in the most recent batch
_pending_eviction:   set[tuple[str, int]]          # (name, lora_id) deferred unloads

_active_names is refreshed at the start of every prepare_loras() call. _pending_eviction accumulates (name, lora_id) pairs when unload_adapter() is called while the adapter may still be in-flight; each entry is flushed on the next batch that does not include that adapter. The tuple form lets _flush_one_pending detect same-name re-registrations and skip GPU eviction if the slot now belongs to a new adapter.

Adapter Lifecycle

This section traces an adapter from first registration through GPU promotion and final unload. The manager separates the durable disk path (recorded at registration) from the transient CPU and GPU residency state (managed through LRU caches), so adapters can be evicted and reloaded transparently.

Key files — python/tokenspeed/runtime/lora/lora_manager.py (added) · python/tokenspeed/runtime/lora/lora_cache.py (added) · python/tokenspeed/runtime/entrypoints/engine.py (modified)
  1. load_adapter(name, path) verifies the adapter weight file or directory.
  2. A new integer lora_id is assigned and stored in _name_to_id and _id_to_name.
  3. The durable path is recorded in the CPU cache object so disk reload remains possible after CPU eviction.
  4. LoraCpuCache.ensure() synchronously loads, parses, and pins weights into the CPU pool when pinned memory is available.
  5. On each forward step, prepare_loras(lora_ids, token_counts) resolves ids to names and then to GPU slots.
  6. If an adapter is CPU-resident but not GPU-resident, _ensure_in_gpu() allocates or evicts a slot and calls _load_to_slot().
  7. _load_to_slot() clears CPU-side metadata (rank, scalings) for the slot, then copies new weights directly into the GPU buffer slice — overwriting any stale values from the previous occupant without a prior GPU zero.
  8. unload_adapter(name) clears GPU slot state, removes CPU cache state, and deletes id mappings.
request lora_id
  -> _id_to_name[lora_id]
  -> _ensure_in_gpu(name)
  -> slot
  -> LoraBatchInfo.weight_indices[segment] = slot

Eviction

Adapter weight eviction is managed entirely in Python by LoraManager. The three-tier design — GPU slots → CPU pinned cache → disk — ensures that evicting from one tier never destroys weights; they are always reloadable from the tier below. Eviction is triggered automatically by prepare_loras() whenever the GPU pool is full and a new adapter is needed.

Key files — python/tokenspeed/runtime/lora/lora_manager.py (added) · python/tokenspeed/runtime/lora/lora_cache.py (added)

How it works — normal path

Every call to prepare_loras(lora_ids, ...) calls _ensure_in_gpu(name) for each unique adapter in the batch. The method follows this sequence:

  1. Already on GPU? — name in _name_to_slot → return the existing slot immediately. No eviction, no disk I/O.
  2. Find a free GPU slot. _find_free_slot() scans _slot_to_name for an empty entry first. If the pool is full it evicts the LRU adapter:
    • Iterate _gpu_lru from front (LRU) to back (MRU).
    • Call _evict_by_name(candidate): removes from _name_to_slot, sets _slot_to_name[slot] = None, zeros CPU-side metadata (rank, scalings), removes from _gpu_lru. GPU weight tensors are not zeroed — stale values are never read because weight_indices never references an empty slot.
    • Return the freed slot.
  3. Promote from CPU to GPU. _cpu_store.ensure(name) guarantees the weights are in CPU pinned memory (loading from disk if needed). _load_to_slot(name, slot) copies each weight tensor to the GPU buffer slice for this slot.
  4. Register and mark MRU. _name_to_slot[name] = slot, _slot_to_name[slot] = name, _gpu_lru[name] = None (appended at end = MRU).

After promoting all unique batch adapters, prepare_loras calls _gpu_lru.move_to_end(name) for each one (phase 2 of the two-phase design) to protect batch adapters from being evicted by a subsequent _ensure_in_gpu call within the same step.

CPU Pool

The CPU pool is bounded by max_loras_cpu (default 4× max_loras). It holds parsed, pinned-memory weight tensors so that GPU promotion is a single non-blocking H2D copy rather than a safetensors parse from disk.

  • prefetch(name) starts a background disk read so the next ensure(name) finds the weights already in CPU memory.
  • ensure(name) blocks until a pending load finishes, or loads synchronously if no prefetch is in flight.
  • CPU eviction prefers adapters that are not currently GPU-resident (those can be reloaded from CPU without a disk read).
  • If no CPU-evictable entry exists (all CPU-resident adapters are also GPU-resident and the pool is at capacity), loading raises RuntimeError.

Explicit unload

unload_adapter(name) removes the adapter from both tiers and blocks future requests from using it. Because GPU kernels read weights asynchronously, unload uses a deferred eviction pattern:

  1. Remove _name_to_id[name] immediately so no new request is assigned this adapter.
  2. If the adapter was active in the most recent forward step (name in _active_names), add (name, lora_id) to _pending_eviction and log a warning. The GPU slot and CPU weights are preserved until a batch arrives that does not include this adapter.
  3. Otherwise, evict the GPU slot and remove CPU weights immediately.
  4. The deferred entry is flushed at the top of the next prepare_loras call for any batch that does not include the adapter.
GPU eviction does not destroy the adapter registration. lora_id and the disk path are preserved so the adapter can be promoted again later without re-registering. Only unload_adapter() permanently removes the registration.

Edge cases

ScenarioBehaviour
Batch needs more unique adapters than max_loras RuntimeError raised at the start of prepare_loras before any eviction occurs. The error is clear and actionable rather than silently producing wrong outputs.
Adapter A is LRU but also needed by the current batch Phase 1 of prepare_loras calls _ensure_in_gpu for all batch adapters first and calls _gpu_lru.move_to_end after each promotion, so A is MRU before phase 2 begins. _find_free_slot evicts non-batch adapters only.
unload_adapter called while adapter is mid-decode Eviction deferred into _pending_eviction. GPU weights and CPU copy kept alive so retracted requests can still resume with correct weights. Eviction fires on the next batch that does not include this adapter.
Same adapter name re-registered before deferred eviction fires _pending_eviction stores (name, old_lora_id) tuples. _flush_one_pending checks whether _name_to_id[name] still matches the recorded old_lora_id; if the name was re-registered with a new id, the GPU slot is left untouched.
LRU eviction races with an in-flight GPU graph kernel reading the slot GPU weight tensors are never zeroed during eviction. The slot is marked logically empty by removing it from _name_to_slot (CPU-only operation). Stale GPU values are overwritten when _load_to_slot reuses the slot. No CUDA stream ordering is required because no GPU write occurs at eviction time.
CPU pool has no evictable entry (all CPU adapters also on GPU) RuntimeError from LoraCpuCache. The server should increase max_loras_cpu or reduce concurrent adapter diversity.

GPU Buffers

Dense LoRA weights are packed into fixed-size per-layer buffers. The first dimension is always n_slots, so kernels can select the active adapter by slot without changing pointer addresses. --lora-buffer-groups controls which coarse families are allocated: attn, mlp, and moe.

The default is attn,mlp,moe. If a server starts with a group disabled, loading an adapter that targets that group raises a configuration error instead of silently dropping LoRA deltas.

qkv_A_buffers[layer](n_slots, 3 * max_rank, hidden)Q, K, V A matrices stacked by rank block.
qkv_B_buffers[layer](n_slots, q_per_tp + 2 * kv_per_tp, max_rank)Column-parallel output side, sharded per TP rank.
o_A_buffers[layer](n_slots, max_rank, o_in_per_tp)Row-parallel input side, sharded along input dimension.
o_B_buffers[layer](n_slots, hidden, max_rank)Replicated output side.
gate_up_A_buffers[layer](n_slots, 2 * max_rank, hidden)Gate and up A matrices stacked.
gate_up_B_buffers[layer](n_slots, 2 * intermediate_per_tp, max_rank)Column-parallel gate/up output side.
down_A_buffers[layer](n_slots, max_rank, intermediate_per_tp)Row-parallel down input side.
down_B_buffers[layer](n_slots, hidden, max_rank)Replicated down output side.

TP Sharding Rule

  • Column-parallel projections (q/k/v, gate, up) shard lora_B along output dimension.
  • Row-parallel projections (o, down) shard lora_A along input dimension.
  • Sharding happens on CPU before the H2D copy, so each TP rank copies only its local shard into GPU buffers.
  • Downstream all-reduce sums base partials and LoRA partials together for row-parallel outputs.

CUDA Graph

The CUDA graph design relies on stable pointers. Adapter contents, segment lengths, slot ids, ranks, and scalings can change between replays, but the tensors holding those values do not move.

Key files — python/tokenspeed/runtime/execution/cuda_graph_wrapper.py (modified)

Capture

  • When LoRA is enabled, CudaGraphWrapper.capture() captures two graphs per batch size.
  • The with-LoRA graph sets ctx.lora_manager and calls prepare_loras([0] * bs) before capture so metadata tensors contain NO_LORA_SLOT while kernels capture stable pointers.
  • The no-LoRA graph leaves ctx.lora_manager unset, so model-layer branches skip LoRA calls entirely.
  • No-LoRA capture is safe because base-model dummy ids resolve to NO_LORA_SLOT; runtime LoRA paths skip work when no real adapter is active.

Replay

  1. ModelExecutor builds the real lora_ids list for the scheduled requests.
  2. prepare_loras() updates the persistent LoraBatchInfo tensors in place.
  3. If any id is nonzero, ctx.lora_manager is set and LoRA-capable layers call apply methods.
  4. CudaGraphWrapper chooses the no-LoRA graph if has_active_lora is false, otherwise it replays the with-LoRA graph.
  5. The captured kernels read the updated metadata and use the current slot-to-weight buffers.
capture time:
  batch_info tensors allocated once
  graph records pointers to batch_info, ranks, scalings, and weight buffers

replay time:
  prepare_loras() mutates tensor contents
  graph.replay() reads new contents through old pointers

Why Two Graphs?

The with-LoRA graph includes LoRA kernel launches. That is necessary when any request uses an adapter. For all-base batches, the no-LoRA graph avoids those launches entirely and preserves base-model decode performance.

Kernel Routing

This section covers how the runtime selects and dispatches the correct Triton kernel for each forward step. The dispatch table balances throughput (prefill SGMV, batched CuTeDSL) against correctness for all segment lengths, slot counts, rank values, and scaling configurations that can appear in a mixed-adapter batch.

Key files — tokenspeed-kernel/python/tokenspeed_kernel/ops/lora/triton/lora_shrink.py (added) · lora_expand.py (added) · lora_qkv_expand.py (added) · lora_gate_up_expand.py (added) · lora_shrink_prefill.py (added) · lora_expand_prefill.py (added) · lora_expand_grouped_v2.py (added)

Dense LoRA applies in two logical phases:

  1. Shrink: compute lora_a = A @ x using the active slot's A buffer.
  2. Expand: compute and add B @ lora_a * scaling into the base layer output.
ConditionPath
max_len > 32Prefill-style shrink/expand kernels.
Decode with grouped slotsGrouped expand path batches tokens by adapter slot.

MoE LoRA

MoE LoRA is deliberately separated from dense buffers. The manager owns MoeLoraBuffers, and MoE backends receive a narrow MoeLoraContext instead of depending on the full LoraManager.

Internal MoE Buffers

MoE LoRA now mirrors the dense/vLLM-style slot model: buffers are preallocated per layer with leading dimensions (n_slots, num_experts, ...). Loading an adapter writes into the selected slot; weights_by_layer[layer][slot] stores views into those fixed buffers for backend consumption. 

w13_A_buffers[layer]:  (n_slots, num_experts, 2 * max_rank, hidden)
w13_B_buffers[layer]:  (n_slots, num_experts, 2 * moe_intermediate_per_tp, 2 * max_rank)
down_A_buffers[layer]: (n_slots, num_experts, max_rank, moe_intermediate_per_tp)
down_B_buffers[layer]: (n_slots, num_experts, hidden, max_rank)

weights_by_layer[layer_id][slot] = {
  "w13_A":  w13_A_buffers[layer_id][slot],
  "w13_B":  w13_B_buffers[layer_id][slot],
  "down_A": down_A_buffers[layer_id][slot],
  "down_B": down_B_buffers[layer_id][slot],
}

When a slot is evicted, only CPU-side metadata (rank, scalings) is cleared. GPU weight tensors are left in place and overwritten by _load_to_slot() when the slot is next assigned. A kernel can only read a slot that weight_indices points to, and evicted slots are immediately removed from _name_to_slot so they are never assigned — stale GPU values are unreachable.

With --lora-moe-compressed-shared-outer, MoE allocation switches to the 3D shared-outer layout instead of full expansion: 

w13_A_buffers[layer]:  (n_slots, 1, 2 * max_rank, hidden)
w13_B_buffers[layer]:  (n_slots, num_experts, 2 * moe_intermediate_per_tp, 2 * max_rank)
down_A_buffers[layer]: (n_slots, num_experts, max_rank, moe_intermediate_per_tp)
down_B_buffers[layer]: (n_slots, 1, hidden, max_rank)

This compressed mode supports shared-outer 3D adapters (w1/w3 A shared, w1/w3 B per-expert, w2 A per-expert, w2 B shared). It rejects per-expert and 2D MoE adapters because those require full expert storage for every side.

Shared-Outer MoE Contract

The 3D shared-outer layout follows the hybrid MoE-LoRA design from Together's research notes. The low-rank side that builds a compact representation can be shared when the representation is common across experts, while the side that interprets an expert-specific activation remains per expert.

In full mode, TokenSpeed expands any dim0=1 shared tensor into every expert slot during load. In compressed mode, the shared side stays physically shared in the GPU pool and MoeLoraContext._select_expert_weights() broadcasts it at apply time. This saves (num_experts - 1) * rank * (3 * hidden) elements per adapter slot per MoE layer, because only w13_A and down_B stop carrying duplicate expert copies.

Route-Level Math

For each routed pair (token t, expert e) and adapter slot s, MoE LoRA adds deltas at the same points as the base MoE projections. Gate/up LoRA is added before the activation; down LoRA is multiplied by the router weight before it is accumulated into the final routed output. 

gate_up_delta[t, e] =
  ((hidden[t] @ w13_A[s, e].T) @ w13_B[s, e].T) * scaling[s]

gate_up_output[t, e] += gate_up_delta[t, e]

down_delta[t, e] =
  ((intermediate[t, e] @ down_A[s, e].T) @ down_B[s, e].T)
  * topk_weights[t, e] * scaling[s]

down_output[t, e] += down_delta[t, e]

When a side is shared, the effective expert index is 0 for that side. The apply path therefore uses the same equations for full per-expert and shared-outer adapters; only the tensor selection changes.

Prefix Cache Namespace

Without LoRA namespacing, two adapters sharing the same prompt prefix would reuse each other's KV pages — producing wrong outputs. The solution embeds a sentinel-key virtual root for each adapter directly inside the shared radix tree. No separate tree or separate allocator is needed.

Virtual Root Structure

KVPrefixCache holds a map lora_virtual_roots_: unordered_map<int32_t, TreeNode*>. When adapter A first inserts tokens, getOrCreateLoraRoot(lora_id) creates a new TreeNode as a child of the real tree root, keyed by a sentinel token vector [-lora_id, 0, 0, …] of length page_size. Because vocabulary tokens are always non-negative, a negative first token can never collide with real content.

The virtual root is given an empty DeviceResource so LRU eviction never prunes it while the adapter is loaded. All of the adapter's KV pages live in the subtree below this node — completely disjoint from the base-model subtree and every other adapter. 

// getOrCreateLoraRoot (kv_prefix_cache.cpp)
token_vec_t sentinel(page_size_, 0);
sentinel[0] = -lora_id;               // unique negative key per adapter
auto node = std::make_unique<TreeNode>(sentinel);
node->AttachResource(make_device_resource(...));  // pin against LRU prune
tree_.Root()->AddChild(sentinel, std::move(node));
lora_virtual_roots_[lora_id] = raw_ptr;

Match: namespace-scoped lookup

Match(token_ids, lora_id) calls resolveStartNode(lora_id) which returns the adapter's virtual root (or nullptr for the base model). WalkDownUntilMismatch() then walks from that start node, so a hit can only return pages that were inserted under this adapter.

The matched depth is adjusted by namespace_depth_offset = 1 to subtract the sentinel page, so callers see the depth as if the virtual root doesn't exist.

Insert: pages land in the right subtree

Insert(token_ids, …, lora_id) likewise calls resolveStartNode(lora_id) before walking. New nodes and their KV pages are attached only within the adapter's subtree. The base model and other adapters are never touched.

Eviction on Unload

The C++ scheduler manages KV prefix-cache page eviction for LoRA adapters — this is entirely separate from LoRA weight eviction handled by Python. When an adapter is unloaded, its virtual radix-tree subtree (and all the KV pages pinned under it) must be freed so the pages can be reused for other requests. The scheduler also performs ongoing LRU eviction of individual KV pages under any adapter's subtree when the device page pool runs low.

How it works — normal LRU eviction

KV page eviction happens continuously as the device page pool fills up. The scheduler's LRU machinery is adapter-oblivious: it evicts the least-recently-used TreeNode regardless of which adapter's virtual root it lives under. The steps are identical for base-model and LoRA subtrees:

  1. Device page pool runs low; EnsureCapacityByEvict(n_pages) is called.
  2. Walk the global LRU list of locked device nodes (oldest first).
  3. For each candidate: detach device resource (frees KV pages), trigger OnKVEvict(node) which releases Mamba slots and detaches any paged-cache snapshot.
  4. If the node is the virtual root of a LoRA adapter, DetachPagedCacheSnapshotFromNode removes it from paged_cache_snapshot_nodes_.

No adapter-specific code runs during normal LRU eviction. An adapter's virtual root and its subtree are evicted the same way as any other node.

How it works — explicit unload via EvictLoraNamespace

When Python calls LoraManager.unload_adapter(name), after deferring or immediately evicting the GPU weights, it sends EvictLoraNamespace(lora_id) to the C++ scheduler. KVPrefixCache::EvictLoraNamespace(lora_id) atomically prunes the entire subtree:

  1. Find virtual root — look up lora_virtual_roots_[lora_id]. If missing or already pruned, return immediately (idempotent).
  2. Collect all descendants — DFS over the subtree to build a flat list of every TreeNode* below the virtual root.
  3. Evict device and host pages — device_.EvictSubtree(descendants) and host_.EvictSubtree(descendants) trigger the normal eviction callbacks on each node: Mamba slot release, paged-cache snapshot detach, KV page deallocation — the exact same path as LRU pressure.
  4. Remove the virtual root — tree_.Root()->RemoveChild(sentinel_key) transfers ownership out via unique_ptr. The cascade destructor walks every remaining node and destroys all attached resources.
  5. Erase from map — lora_virtual_roots_.erase(lora_id). The adapter namespace is gone; subsequent Match calls for this lora_id will call getOrCreateLoraRoot and rebuild from scratch.
No adapter-specific eviction code exists. The virtual root is a normal TreeNode, so LRU ordering, Mamba L2 writeback, and paged-cache snapshot detach all run identically for LoRA and base-model subtrees.

Gaps & Status

Two of the original five gaps have been fixed. Three performance gaps remain open.

✅ Fixed

1 — Adapter-aware co-scheduling

Was: newForwardOperation() ignored lora_id, allowing batches requiring more unique adapters than max_loras. When prepare_loras() then evicted one batch adapter to promote another, the evicted adapter's requests silently fell back to NO_LORA_SLOT and ran as the base model — wrong outputs with no error raised.

Fix: max_loras is now a field on SchedulerConfig, set from server_args.max_loras. newForwardOperation() maintains a batch_lora_ids set and defers any request whose lora_id would push the count past the cap. As a side-effect, requests for the same adapter cluster into the same step, reducing GPU slot thrashing. 

// scheduler/operations/forward.cpp  (forward.h, types.h, bindings)
std::unordered_set<int32_t> batch_lora_ids;
for (Request* request : candidates) {
    if (config_.max_loras > 0 && request->lora_id() != kLoraNone) {
        bool is_new = !batch_lora_ids.count(request->lora_id());
        if (is_new && (int)batch_lora_ids.size() >= config_.max_loras)
            continue;   // defer to next step
        batch_lora_ids.insert(request->lora_id());
    }
    // ... schedule request normally
}

2 — GPU slot exhaustion crash

Was: if every max_loras GPU slot was held by an in-flight adapter, _find_free_slot() raised RuntimeError: LoRA GPU pool is full and no evictable adapter was found mid-batch, surfacing as a batch failure instead of a clean deferral.

Fix (consequence of Gap 1): the scheduler now guarantees n_unique ≤ max_loras before building the batch, so the pool always has room for every adapter. The crash path in _find_free_slot() is unreachable under normal operation. prepare_loras() still raises a clear error if the invariant is violated externally (defence-in-depth).

🔲 Open — performance only

3 — No per-adapter KV page accounting

KV page eviction is global LRU across all adapters and the base model. A hot adapter can lose its cached prefix pages to a one-shot base-model request, forcing full KV recomputation on the next step. The simulated-free admission tracking operates per paged-cache group, not per adapter.

Where to fix: add an adapter-priority weight to the LRU eviction comparator in KVPrefixCache, or a minimum-pages reservation per active adapter slot in SchedulerConfig.

4 — No KV prefix warming signal

The scheduler can issue a PrefetchOperation to move KV pages from host to device before a request runs, but there is no adapter-level counterpart: common system prompts cannot be pre-populated into the KV prefix cache at adapter load time. The first request for a freshly loaded adapter always incurs a cold-start KV computation cost.

Where to fix: at load_lora_adapter() time, submit a synthetic warmup request with the adapter's canonical system prompt so the scheduler fills the prefix cache before real traffic arrives. No C++ changes required.

5 — KV eviction does not distinguish active from idle adapters

An adapter with no in-flight requests has its KV prefix pages compete on equal terms in the global LRU. A burst of base-model or other-adapter traffic can evict all of an adapter's prefix pages during a quiet period, so the next request for that adapter starts cold even though the adapter is still loaded and ready.

Where to fix: introduce an LRU priority weight per adapter — e.g., boost the last-access time of an adapter's virtual root each time it serves a request. Alternatively, reserve a minimum number of KV pages per loaded adapter slot in SchedulerConfig.
